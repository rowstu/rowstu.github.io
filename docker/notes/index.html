Docker Notes

Home / Browse/ Dockernotes

Docker is a platform that automates the deployment of applications inside lightweight, portable containers. Containers package an application with everything it needs to run: code, runtime, libraries, and system tools.

Key Benefits

  • Consistency: Runs the same way everywhere (dev, test, prod).
  • Isolation: Containers run independently and don’t interfere with each other.
  • Efficiency: Uses fewer resources than VMs (no full OS per container).
  • Portability: Works across different OSes and environments.
  • Speed: Containers start in seconds.

Core Concepts

Containers

  • Lightweight, standalone runtime environments.
  • Run processes in isolation.
  • Share the host kernel but have isolated user space.

Images

  • Read-only templates used to create containers.
  • Built from layers (each command in Dockerfile creates a new layer).
  • Can be versioned/tagged.

Dockerfile

  • Text file defining how to build an image.
  • Contains instructions like FROM, RUN, COPY, CMD.

Registry

  • Storage for Docker images (e.g., Docker Hub, GitHub Container Registry).
  • Allows sharing and distributing images.

Docker Engine

  • Runtime that builds and runs containers.

Installation

Linux (Ubuntu)

sudo apt update
sudo apt install -y docker.io
sudo systemctl enable --now docker
sudo usermod -aG docker $USER

Linux (CentOS/RHEL)

sudo yum install -y yum-utils
sudo yum-config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
sudo yum install docker-ce docker-ce-cli containerd.io
sudo systemctl enable --now docker
sudo usermod -aG docker $USER
  • Verify installation:
docker version
docker info
  • Test run:
docker run hello-world

3. Basic Commands & Usage

Common Commands

Run a container

docker run -it ubuntu bash

List containers

docker ps
docker ps -a

Stop / Start / Remove

docker stop <container_id>
docker start <container_id>
docker rm <container_id>

Pull an image

docker pull nginx:latest

Build an image

docker build -t myapp:1.0 .

List images

docker images

Basic Workflow Example

1. Create Dockerfile

FROM python:3.12-slim
WORKDIR /app
COPY . /app
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

2. Build image

docker build -t myapp:1.0 .

3. Run container

docker run -d -p 5000:5000 myapp:1.0

4. Verify

curl http://localhost:5000

4. Docker Networking

Networking Principles

  • Each container gets its own network namespace.
  • Containers can communicate via virtual network interfaces.
  • Docker creates a default bridge network (docker0).

Network Types

  • Bridge (default): Containers on same host communicate via bridge.
  • Host: Container uses host’s network stack.
  • Overlay: Multi-host networking (Docker Swarm).
  • Macvlan: Containers get IPs from LAN subnet.

Create & Manage Networks

docker network create mynet
docker network ls
docker network inspect mynet

Run containers on same network

docker run -d --name app1 --network mynet nginx
docker run -d --name app2 --network mynet nginx

Communication

docker exec -it app1 ping app2

Managing Docker Images

Best Practices

  • Use official base images (e.g., python, nginx, alpine)
  • Keep images small (avoid bloated base images)
  • Pin versions (python:3.12-slim, not python:latest)
  • Minimize layers (combine commands)
  • Use .dockerignore to exclude unnecessary files

Optimizing Image Size

  • Use multi-stage builds
  • Remove build dependencies after install
  • Use slim or alpine base images
  • Clean package caches

Example (multi-stage build)

FROM node:20 AS builder
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/build /usr/share/nginx/html

Docker Compose

What is Docker Compose?

Docker Compose defines and runs multi-container applications using a YAML file (docker-compose.yml).

Benefits

  • Orchestrate multi-container apps easily
  • Simplify complex setups (DB + app + cache)
  • Manage lifecycle with single commands

Example Compose File

version: "3.9"
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: example
    volumes:
      - dbdata:/var/lib/postgresql/data

volumes:
  dbdata:

Commands

docker compose up -d
docker compose ps
docker compose logs -f
docker compose down

Compose Best Practices

  • Use environment variables for secrets/config
  • Use named volumes for persistent data
  • Separate dev and prod compose files
  • Keep services small and single-purpose

Storage & Volumes

Data Management

Containers are ephemeral by default: when removed, data is lost.

Two main options

  • Volumes (recommended): Managed by Docker
  • Bind mounts: Directly map host filesystem

Volumes

docker volume create mydata
docker run -d -v mydata:/data myapp

Best Practices

  • Use volumes for databases and persistent storage
  • Avoid storing data inside container layers
  • Backup volumes regularly

Bind Mounts

docker run -v /host/path:/container/path myapp

Use Cases

  • Local development
  • Sharing host config files

Security Best Practices

Key Considerations

  • Containers share the host kernel → vulnerabilities matter.
  • Running containers as root is risky.
  • Images from untrusted sources can contain malware.

Recommendations

  • Run as non-root
RUN useradd -ms /bin/bash appuser
USER appuser
  • Scan images for vulnerabilities (e.g., Trivy)
  • Limit container capabilities
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE ...
  • Use read-only filesystem
docker run --read-only ...
  • Use secrets management (not env vars)
  • Isolate networks and limit exposed ports
  • Keep Docker daemon access restricted

Troubleshooting & Maintenance

Common Issues

Container won’t start

  • Check logs:
docker logs <container_id>

Port already in use

  • Check running containers:
docker ps
  • Stop conflicting container or change port mapping.

Image build failing

  • Check Dockerfile syntax
  • Clear cache and rebuild:
docker build --no-cache -t myapp .

Maintenance Best Practices

  • Regularly prune unused resources:
docker system prune -a
  • Clean up dangling images:
docker image prune
  • Monitor disk usage:
docker system df
