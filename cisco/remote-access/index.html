Configuring Remote Access to Cisco Switches

Complete guide to setting up SSH and secure remote management access on Cisco switches.

Overview

Remote management allows you to configure and monitor switches without physical console access. This guide covers:

  • Assigning management IP addresses
  • Configuring SSH (recommended)
  • Securing remote access
  • Troubleshooting connectivity

Prerequisites

Before configuring remote access:

  • Console access to the switch
  • Network connectivity to the switch
  • IP address plan for management interface
  • DNS/hostname (required for SSH)

Assigning an IP Address to a Switch

Method 1: Using VLAN 1 (Default, Simple)

Switch# configure terminal

! Configure VLAN 1 interface
Switch(config)# interface vlan 1
Switch(config-if)# ip address 192.168.1.10 255.255.255.0
Switch(config-if)# no shutdown
Switch(config-if)# description Management Interface
Switch(config-if)# exit

! Set default gateway
Switch(config)# ip default-gateway 192.168.1.1

! Save configuration
Switch(config)# exit
Switch# write memory

Method 2: Using Dedicated Management VLAN (Best Practice)

! Create management VLAN
Switch(config)# vlan 99
Switch(config-vlan)# name MANAGEMENT
Switch(config-vlan)# exit

! Configure VLAN interface
Switch(config)# interface vlan 99
Switch(config-if)# ip address 192.168.99.10 255.255.255.0
Switch(config-if)# no shutdown
Switch(config-if)# description Management Interface
Switch(config-if)# exit

! Set default gateway
Switch(config)# ip default-gateway 192.168.99.1

! Assign a physical port to management VLAN (for connectivity)
Switch(config)# interface gigabitethernet 0/24
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 99
Switch(config-if)# description Management Port
Switch(config-if)# exit

Verify IP Configuration

Switch# show ip interface brief
Interface              IP-Address      OK? Method Status                Protocol
Vlan1                  unassigned      YES unset  administratively down down
Vlan99                 192.168.99.10   YES manual up                    up

! Test connectivity
Switch# ping 192.168.99.1

Configuring SSH Access

SSH provides encrypted remote access. Telnet is not recommended (unencrypted).

Step 1: Set Hostname and Domain Name

Required for SSH key generation:

Switch(config)# hostname CoreSwitch
CoreSwitch(config)# ip domain-name company.local

Step 2: Generate RSA Keys

CoreSwitch(config)# crypto key generate rsa
The name for the keys will be: CoreSwitch.company.local
Choose the size of the key modulus in the range of 360 to 4096 for your
  General Purpose Keys. Choosing a key modulus greater than 512 may take
  a few minutes.

How many bits in the modulus [512]: 2048
% Generating 2048 bit RSA keys, keys will be non-exportable...
[OK] (elapsed time was 3 seconds)

Key sizes:

  • 1024 bits: Minimum for SSH version 2 (acceptable)
  • 2048 bits: Recommended
  • 4096 bits: Maximum security (slower)

Step 3: Configure SSH Version 2

CoreSwitch(config)# ip ssh version 2
CoreSwitch(config)# ip ssh time-out 60
CoreSwitch(config)# ip ssh authentication-retries 3

Step 4: Create Local User Accounts

CoreSwitch(config)# username admin privilege 15 secret Admin123!@#
CoreSwitch(config)# username netadmin privilege 15 secret NetAdmin456!@#
CoreSwitch(config)# username readonly privilege 1 secret ReadOnly789!@#

Step 5: Configure VTY Lines

CoreSwitch(config)# line vty 0 15
CoreSwitch(config-line)# transport input ssh
CoreSwitch(config-line)# login local
CoreSwitch(config-line)# exec-timeout 10 0
CoreSwitch(config-line)# logging synchronous
CoreSwitch(config-line)# exit

VTY Line Parameters:

  • transport input ssh - Only allow SSH (block Telnet)
  • login local - Use local user accounts
  • exec-timeout 10 0 - 10-minute session timeout

Step 6: Save and Test

CoreSwitch(config)# exit
CoreSwitch# write memory

! From another device
ssh admin@192.168.99.10

Restricting SSH Access with ACLs

Limit SSH access to specific IP addresses or networks:

Create Access List

! Allow SSH from management network only
CoreSwitch(config)# access-list 10 remark Management Network Only
CoreSwitch(config)# access-list 10 permit 192.168.100.0 0.0.0.255
CoreSwitch(config)# access-list 10 deny any log

Apply to VTY Lines

CoreSwitch(config)# line vty 0 15
CoreSwitch(config-line)# access-class 10 in
CoreSwitch(config-line)# exit

Result: Only devices from 192.168.100.0/24 can SSH to the switch.

Extended ACL Example

! Named extended ACL for more control
CoreSwitch(config)# ip access-list extended SSH-ACCESS
CoreSwitch(config-ext-nacl)# remark Allow SSH from admin workstations
CoreSwitch(config-ext-nacl)# permit tcp host 192.168.100.10 any eq 22
CoreSwitch(config-ext-nacl)# permit tcp host 192.168.100.11 any eq 22
CoreSwitch(config-ext-nacl)# permit tcp 10.0.0.0 0.255.255.255 any eq 22
CoreSwitch(config-ext-nacl)# deny ip any any log
CoreSwitch(config-ext-nacl)# exit

! Apply to VTY
CoreSwitch(config)# line vty 0 15
CoreSwitch(config-line)# access-class SSH-ACCESS in

AAA Authentication for Remote Access

For enterprise environments, use centralized authentication:

TACACS+ Configuration

! Enable AAA
CoreSwitch(config)# aaa new-model

! Configure authentication
CoreSwitch(config)# aaa authentication login default group tacacs+ local
CoreSwitch(config)# aaa authorization exec default group tacacs+ local

! Configure TACACS+ servers
CoreSwitch(config)# tacacs-server host 192.168.100.50
CoreSwitch(config)# tacacs-server host 192.168.100.51
CoreSwitch(config)# tacacs-server key TacacsSecretKey123!

! Apply to VTY lines
CoreSwitch(config)# line vty 0 15
CoreSwitch(config-line)# authorization exec default
CoreSwitch(config-line)# accounting exec default start-stop group tacacs+
CoreSwitch(config-line)# exit

RADIUS Configuration

! Enable AAA
CoreSwitch(config)# aaa new-model

! Configure authentication
CoreSwitch(config)# aaa authentication login default group radius local
CoreSwitch(config)# aaa authorization exec default group radius local

! Configure RADIUS servers
CoreSwitch(config)# radius-server host 192.168.100.60 auth-port 1812 acct-port 1813
CoreSwitch(config)# radius-server key RadiusSecretKey456!

! Apply to VTY lines
CoreSwitch(config)# line vty 0 15
CoreSwitch(config-line)# authorization exec default
CoreSwitch(config-line)# accounting exec default start-stop group radius

Always include local as fallback in case AAA server is unreachable!

Configuring Telnet (Not Recommended)

Telnet transmits passwords in clear text. Only use in isolated lab environments.

Switch(config)# line vty 0 15
Switch(config-line)# transport input telnet
Switch(config-line)# password TelnetPass123
Switch(config-line)# login
Switch(config-line)# exit

To allow both SSH and Telnet:

Switch(config-line)# transport input ssh telnet

Best practice: transport input ssh (SSH only)

HTTP/HTTPS Web Interface

Some Cisco switches support web-based management:

Enable HTTP

Switch(config)# ip http server
Switch(config)# ip http authentication local

Enable HTTPS (Recommended)

Switch(config)# ip http secure-server
Switch(config)# ip http authentication local
Switch(config)# no ip http server

Access via: https://192.168.99.10

Restrict HTTP Access

Switch(config)# ip http access-class 10

Note: Web interface is less common and not recommended for production. Use SSH.

Layer 3 Switch - Routed Port Configuration

For Layer 3 switches with routed ports:

! Enable IP routing
Switch(config)# ip routing

! Configure routed port (not a switchport)
Switch(config)# interface gigabitethernet 0/1
Switch(config-if)# no switchport
Switch(config-if)# ip address 10.1.1.1 255.255.255.0
Switch(config-if)# no shutdown
Switch(config-if)# description WAN Link
Switch(config-if)# exit

! Configure default route (if needed)
Switch(config)# ip route 0.0.0.0 0.0.0.0 10.1.1.254

Securing Remote Access

1. Use SSH Only (Never Telnet)

line vty 0 15
 transport input ssh

2. Implement Access Control Lists

access-list 10 permit 192.168.100.0 0.0.0.255
line vty 0 15
 access-class 10 in

3. Use Strong Passwords

username admin privilege 15 algorithm-type scrypt secret C0mplex!P@ssw0rd#123

4. Enable Session Timeouts

line vty 0 15
 exec-timeout 10 0

5. Limit Concurrent Sessions

line vty 0 4
 ! Only 5 simultaneous SSH sessions (lines 0-4)

6. Enable Logging

CoreSwitch(config)# logging buffered 51200
CoreSwitch(config)# logging console warnings
CoreSwitch(config)# logging monitor informational
CoreSwitch(config)# service timestamps log datetime msec
CoreSwitch(config)# service timestamps debug datetime msec

7. Configure Banner

Legal warning banner:

CoreSwitch(config)# banner login ^C
*******************************************************************************
*                                                                             *
*  WARNING: Unauthorized access to this device is prohibited!                *
*           All activities are monitored and logged.                          *
*           Violators will be prosecuted to the fullest extent of the law.    *
*                                                                             *
*******************************************************************************
^C

Troubleshooting Remote Access

Cannot Connect via SSH

1. Verify IP configuration

Switch# show ip interface brief
Switch# ping [destination]

2. Verify SSH is enabled

Switch# show ip ssh

If SSH is disabled:

Switch(config)# ip ssh version 2

3. Verify RSA keys are generated

Switch# show crypto key mypubkey rsa

If no keys:

Switch(config)# crypto key generate rsa modulus 2048

4. Check VTY configuration

Switch# show running-config | section line vty

5. Verify ACL isn’t blocking

Switch# show access-lists
Switch# show line vty 0 15 | include access-class

SSH Connection Refused

Possible causes:

  • RSA keys not generated
  • SSH not enabled
  • VTY lines not configured for SSH

Fix:

Switch(config)# crypto key generate rsa modulus 2048
Switch(config)# ip ssh version 2
Switch(config)# line vty 0 15
Switch(config-line)# transport input ssh

Password Rejected

1. Verify user account exists

Switch# show running-config | include username

2. Check authentication method

Switch# show running-config | section line vty

If using login local, ensure user exists:

Switch(config)# username admin privilege 15 secret NewPassword123!

Connection Times Out

1. Check network connectivity

ping 192.168.99.10
telnet 192.168.99.10 22

2. Verify default gateway

Switch# show running-config | include ip default-gateway

3. Check interface status

Switch# show ip interface brief
Switch# show interface vlan 99

AAA Server Unreachable

If using TACACS+/RADIUS and server is down:

Ensure local fallback is configured:

Switch# show running-config | include aaa authentication
aaa authentication login default group tacacs+ local

If missing local:

Switch(config)# aaa authentication login default group tacacs+ local

Remote Access Checklist

Pre-Configuration

  • Plan IP addressing scheme
  • Document management VLAN
  • Plan ACL for SSH access
  • Prepare user accounts
  • Test network connectivity

Configuration

  • Hostname configured
  • Domain name configured
  • Management IP assigned
  • Default gateway configured
  • RSA keys generated (2048-bit minimum)
  • SSH version 2 enabled
  • Local user accounts created
  • VTY lines configured for SSH only
  • Access control list created and applied
  • Session timeout configured
  • Logging enabled
  • Banner configured
  • Configuration saved

Testing

  • SSH from authorized network works
  • SSH from unauthorized network blocked
  • Telnet blocked (if applicable)
  • Session timeout works
  • Logging captures SSH sessions
  • User authentication works
  • AAA fallback works (if configured)

Documentation

  • Management IP documented
  • User accounts documented
  • ACL details documented
  • Network diagram updated
  • Password vault updated

Security Best Practices

  1. Always use SSH, never Telnet - encryption is essential
  2. Generate strong RSA keys - 2048 bits minimum
  3. Implement access control lists - restrict source IPs
  4. Use individual user accounts - no shared passwords
  5. Enable session timeouts - 10 minutes maximum
  6. Configure AAA with local fallback - don’t lock yourself out
  7. Use dedicated management VLAN - separate from user traffic
  8. Enable logging - monitor all access
  9. Display legal banner - warn unauthorized users
  10. Keep SSH updated - use latest IOS version with security patches
